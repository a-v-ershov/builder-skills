# builder-skills

[English](README.md) · **Русский**

**Доведите сырую идею до готового релиза вместе с Builder Skills — по одному проверяемому шагу за
раз, чтобы ИИ строил *то, что нужно*, и доказывал, что оно работает, а не просто заявлял об этом.**

Три конвейера с воротами — **спецификация → сборка → релиз** — устанавливаются как плагин в любой
проект прямо с GitHub. Каждый скилл отвечает на том языке, на котором вы к нему обращаетесь.

---

## 3 проблемы, которые он решает

1. **ИИ быстро строит не то** — он прыгает от однострочного запроса сразу к коду, и вы выкатываете отполированную версию непроверенной идеи, а понимаете это слишком поздно.
2. **ИИ сам проверяет свою работу** — один и тот же агент пишет функцию, объявляет *«работает»* и пишет тесты, которые проходят, потому что он написал их так, чтобы они проходили.
3. **Качество и решения испаряются** — дыры в безопасности, медленные места и причины принятых решений никогда не видны в зелёном модульном тесте и теряются между сессиями.

## Что вы получаете

1. **Три конвейера, по одной команде на каждый** — вызовите `create-project-spec`, `build-product` или `release-product`, и один мета-скилл проведёт за вас все фазы.
2. **Сначала он вас расспрашивает, а потом строит** — дотошное интервью вытягивает из вашей головы максимум контекста, вплоть до предпочтений по стеку и стилю, по одному вопросу за раз и всегда с рекомендованным ответом.
3. **Работу всегда проверяет отдельный агент** — ничто не заверяет само себя: ревьюер на каждой фазе спецификации плюс свежий проверяющий, который пишет состязательные тесты, а хук не даёт ему трогать код.
4. **Настоящий цикл разработки + релизные аудиты, выискивающие собственные ошибки ИИ** — обязательные ворота качества локально, затем независимые аудиты безопасности / производительности / качества / доступности / документации перед выпуском.

**Плюс:** отвечает на вашем языке · восстанавливает спецификацию из готового кода (brownfield) ·
оставляет зафиксированную в репозитории память проекта — документы, ADR, бэклог, карту в `CLAUDE.md` —
которая не деградирует · поставляется как обычный плагин Claude Code, без отдельной среды выполнения и
MCP-сервера.

## Как это работает

Три конвейера, каждый — тонкий **оркестратор**, который дирижирует узкоспециализированными
под-скиллами; каждый под-скилл работает и сам по себе.

### 1 — Спецификация: идея → готовая к сборке спецификация

`create-project-spec` проходит семь фаз, у каждой своя роль; каждая исследует свои утверждения (со
ссылками на источники), пишет черновик, проверяется состязательным ревьюером и выдаёт исследовательский
документ + короткое резюме для человека в `docs/project-spec/`.

| Шаг | Скилл | Роль | Производит |
|------|-------|------|------------|
| 1 | `gather-context` | Интервьюер-исследователь | `project-brief` — расспрашивает вас, пока не возникнет общее понимание |
| 2 | `validate-idea` | Основатель-инвестор | `idea-validation` — KILL / SHRINK / неудобные вопросы |
| 3 | `define-product-requirements` | Менеджер продукта | `product-requirements` — полный набор функций, каждая с проверяемыми критериями |
| 4 | `create-user-flows` | Продуктовый дизайнер | `user-flows` — как пользователи движутся по продукту |
| 5 | `define-design-decisions` | Ведущий по дизайн-системе | `design-decisions` — направление, не макеты; мост к технике |
| 6 | `design-architecture` | Архитектор ПО | `architecture` + ADR — сначала сценарии качества, затем компоненты + технологии |
| 7 | `design-dev-architecture` | Инженер DX / платформы | `dev-architecture` + ADR — локальный внутренний цикл и ИИ-инструментарий |

> **Brownfield (готовый код):** направьте его на существующий код (`project_type: existing`), и первым
> запустится `map-codebase`, который зафиксирует фактическое состояние; затем каждая фаза реконструирует
> *целевую* спецификацию и записывает расхождения.

### 2 — Сборка: спецификация → работающее ПО

`build-product` превращает спецификацию в код последовательно — по одной задаче за раз, в едином
рабочем дереве, без параллелизма — изменяя настоящий репозиторий.

| Шаг | Скилл | Роль |
|------|-------|------|
| 1 | `setup-dev-environment` | Разворачивает репозиторий, поднимает стек одной командой, ставит обязательные ворота качества (`make check` + хуки) |
| — | `create-design-system` | *(UI)* делает направление дизайна конкретным — `DESIGN.md`, зафиксированный в репозитории, который вы выбираете из отрисованных вариантов |
| 2 | `plan-development` | Выдаёт канбан-бэклог — по файлу на задачу, где `blocked_by` *и есть* граф зависимостей |
| 3 | `implement-feature` | Реализует одну задачу и доводит ворота качества до зелёного |
| 4 | `verify-feature` | **Свежий, отдельный агент**, который пишет состязательные тесты, гоняет настоящий стек и доказывает наблюдаемый результат |

Он берёт готовую задачу с наименьшим id, реализует её, проверяет отдельным агентом (с ограничением — на
пределе задача эскалируется в `needs_human`) и при зелёных воротах делает контрольную фиксацию.
`generate-mockups` по запросу отрисовывает варианты интерфейса; `propagate-changes` проводит позднюю
правку спецификации вперёд в бэклог.

### 3 — Релиз: работающее ПО → готовый релиз

`release-product` доказывает сквозные свойства, которые не под силу ни одной отдельной задаче, и затем
выпускает; аудиты — только на чтение, поэтому они расходятся **параллельно** и заводят находки как
доработки, а не правят код сами.

| Скилл | Что доказывает |
|-------|----------------|
| `audit-security` | модель угроз STRIDE-lite — секреты, авторизация, инъекции, «смертельная троица» |
| `audit-performance` | сценарии атрибутов качества — измеренные p95, пропускная способность, N+1, стоимость |
| `audit-product` | пользовательские сценарии, насквозь и между функциями |
| `audit-code-health` | дублирование, мёртвый код, долг подавленных предупреждений, качество тестов (мутационное) |
| `audit-accessibility` | цель по WCAG — клавиатура, фокус, скринридер, контраст |
| `cut-release` | чистое дерево + нет открытых 🔴 → документы, версия, журнал изменений, тег, коммит, PR (всегда с подтверждением; **останавливается перед развёртыванием в продакшене**) |

Блокер становится задачей-доработкой, исправляется запуском `build-product`, а затем **повторно
проверяется аудитом, чтобы подтвердить, что он закрыт**.

## Сравнение

Большинство инструментов покрывают лишь один отрезок пути; ставка builder-skills — *весь* путь плюс
отдельный оппонент-проверяющий на каждом шаге.

| Инструмент | Что это | Чем отличается builder-skills |
|------|---------|-------------------------------|
| **[GitHub Spec Kit](https://github.com/github/spec-kit)** | Spec → Plan → Tasks → Implement; шаблоны, не привязан к агенту | Останавливается на реализации — нет независимой проверки, релизных аудитов, встроенного исследования и состязательного ревью. |
| **[BMAD-METHOD](https://github.com/bmad-code-org/BMAD-METHOD)** | Agile-фреймворк на агентах, 12+ специализированных агентов | Планирует и собирает, но нет фазы релизных аудитов только на чтение, нет проверяющего на уровне харнесса, нет реконструкции из готового кода. |
| **[Task Master](https://github.com/eyaltoledano/claude-task-master)** | PRD → разбиение на задачи; MCP-менеджер проекта | Только управление задачами — не валидирует, не исследует, не проектирует и не аудирует; требует запущенного MCP-сервера. |
| **[gstack](https://github.com/garrytan/gstack)** | 23 ролевых скилла для цикла «на каждый PR» | Хорош в цикле ревью/выпуска на готовом коде, но без генератора «идея → спецификация» и без реконструкции спецификации. *(Наш ориентир по дизайну.)* |
| **[SuperClaude](https://github.com/SuperClaude-Org/SuperClaude_Framework)** | ~30 команд + ролевые агенты, внедряемые в `CLAUDE.md` | Набор команд «по меню», а не последовательный конвейер с воротами, в который встроены исследование, ревью и разделение «автор/ревьюер». |

## Установка

В любом проекте, внутри Claude Code:

```
/plugin marketplace add https://github.com/a-v-ershov/builder-skills.git
/plugin install builder-skills@builder-skills
```

Скиллы появляются с префиксом `builder-skills:<skill>`. Начните проект с
**`/builder-skills:create-project-spec`** и ответьте на три вопроса настройки — или запускайте любой
отдельный скилл сам по себе.

## Обновление

Установка подхватывает изменения только после **повышения** версии плагина (`version`). Чтобы обновить
проект, где он уже установлен (обновление применяется после перезапуска):

```
/plugin marketplace update builder-skills
/plugin update builder-skills@builder-skills
```

Или включите автообновление один раз: `/plugin` → **Marketplaces** → `builder-skills` →
**Enable auto-update**.

Все 25 скиллов, сгруппированы по конвейерам. Оркестраторы идут первыми в каждой группе; каждый
под-скилл работает и сам по себе.

### Спецификация — идея → готовая к сборке спецификация

| `map-codebase` | Археолог кода | *(существующий)* Реконструирует фактическое состояние готовой кодовой базы | `codebase-map.research.md` |
| `gather-context` | Интервьюер-исследователь | Расспрашивает вас, превращая короткий бриф в общее понимание | `project-brief.research.md` |
| `validate-idea` | Основатель-инвестор | Проверяет на прочность спрос, аудиторию, проблему и бизнес-модель | `idea-validation.research.md` |
| `define-product-requirements` | Менеджер продукта | Определяет, для кого продукт, и полный зафиксированный набор функций + критерии | `product-requirements.research.md` |

| `design-architecture` | Архитектор ПО | Сначала сценарии качества, затем компоненты + реализующие их технологии | `architecture.research.md` + `adr/` |
| `design-dev-architecture` | Инженер DX / платформы | Проектирует локальный внутренний цикл, тестирование под управлением ИИ и ИИ-инструментарий | `dev-architecture.research.md` + `adr/` |

### Сборка — спецификация → работающее ПО

| Скилл | Роль | Что делает | Пишет |
|-------|------|------------|-------|
| `build-product` | Оркестратор | Берёт по одной готовой задаче за раз и ведёт цикл реализация → проверка → коммит | цикл сборки |
| `setup-dev-environment` | Инженер платформы | Разворачивает репозиторий, поднимает стек, ставит ворота качества | репозиторий + `docs/project-setup/` |
| `create-design-system` | Ведущий по дизайн-системе | *(UI)* Делает направление дизайна конкретным из отрисованных вариантов | `DESIGN.md` + `design-system.md` |
| `plan-development` | Ведущий разработки | Превращает спецификацию в канбан-бэклог, по файлу на задачу | `docs/build-plan/` |
| `implement-feature` | Разработчик | Реализует одну задачу из бэклога и доводит ворота качества до зелёного | код |
| `verify-feature` | Независимый проверяющий | Пишет состязательные тесты и доказывает наблюдаемые результаты собранной задачи | тесты + вердикт |

### Релиз — работающее ПО → готовый релиз

| Скилл | Роль | Что делает | Пишет |
|-------|------|------------|-------|
| `release-product` | Капитан релиза | Запускает аудиты, заводит доработки, ведёт исправления, перепроверяет и затем выпускает | цикл релиза |
| `audit-security` | Инженер по безопасности | Доказывает модель угроз STRIDE-lite на работающей системе | `security-audit.md` |
| `audit-performance` | Инженер по производительности | Измеряет систему против сценариев атрибутов качества | `performance-audit.md` |
| `audit-product` | Ведущий QA | Проводит пользовательские сценарии насквозь, между функциями | `qa-report.md` |
| `audit-code-health` | Старший инженер | Измеряет деградацию кода, долг подавлений и качество тестов | `code-health-audit.md` |
| `audit-accessibility` | Специалист по доступности | *(UI)* Гоняет интерфейс против цели по WCAG | `accessibility-audit.md` |
| `cut-release` | Релиз-инженер | Повышает версию, журнал изменений, тег, коммит, PR — с воротами, останавливается перед развёртыванием | релизные документы + PR |

### Сквозные — используются по всем конвейерам

| Скилл | Роль | Что делает | Пишет |
|-------|------|------------|-------|
| `commit` | Помощник по git | Разбивает изменения сессии на аккуратные коммиты (сообщения на английском) | коммиты |
| `propagate-changes` | Сквозной дирижёр | Согласует нижестоящие документы + бэклог после правки вышестоящей спецификации | обновлённые документы + бэклог |

К каждому `*.research.md` прилагается парный `*.summary.md`; документы спецификации лежат в
`docs/project-spec/`, аудиты — в `docs/release/`. *(существующий)* = только проекты brownfield,
*(UI)* = только проекты с интерфейсом (иначе пропускается), *(по запросу)* = не запускается автоматически
в конвейере. `gather-context` также можно вызвать по запросу как точечное интервью.
